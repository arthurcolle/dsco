This should not be designed as a thin reverse proxy. 1. Core design principles

These are the architectural constraints that should shape every endpoint and backend service.

01Compatibility is a product feature

/v1/* must stay stable enough that Claude-oriented clients can be pointed at api.distributed.systems without rewriting their request semantics.

02Native DSCO APIs should not hide behind Anthropic shapes

The DSCO product surface should live under /ds/v1/*. Mixing native constructs into /v1/* creates long-term compatibility debt.

03Artifact memory is a first-class plane

Config, debug logs, memory files, runtime transcripts, and generated outputs are not incidental logs. They are retrievable planning state.

04Async by default

Long-running swarm operations should never depend on the lifetime of a single HTTP request. Every mutating execution endpoint needs resumable run state.

05SQLite first is the right decision

Use SQLite with WAL, strict schemas, structured event tables, and per-workspace replication. Move off only when the actual bottleneck is measured.

06Streaming must survive noise

SSE, chunked responses, tool bursts, MCP chatter, and large result bodies should be handled without timeout-driven kill paths.

2. Public API surface

The edge should expose three distinct categories of routes with different stability promises.

Compatibility API

Anthropic-compatible routes under /v1/*. This is the conservative facade. Behavior should track upstream closely enough to preserve client expectations.

Claude transport API

Observed Claude-specific endpoints like MCP server discovery, MCP session transport, and event logging. These sit adjacent to compatibility routes because clients already expect them.

DSCO native API

Swarm control plane, topology planning, artifacts, embeddings, telemetry, policy, and run management under /ds/v1/*. This is where product differentiation belongs.

3. Compatibility facade

Exact endpoint parity is not the goal on day one. Behavioral parity on the high-value execution paths is.

AExecution endpoints
POST /v1/messages

Primary compatibility path. Must support streaming, tool use, long outputs, and precise header pass-through.

POST /v1/messages/count_tokens

Needed for cost estimation, context planning, and client-side request shaping.

POST /v1/messages/batches

Async batch control. Internally this should map to DSCO run groups instead of a blind upstream proxy.

BStatic and admin endpoints
GET /v1/models

Expose both pass-through upstream models and DSCO-routed synthetic model aliases if needed.

GET /v1/files

Compatibility route for file-backed prompting and artifacts. This should map into the DSCO artifact store internally.

GET /v1/organizations/*

Admin parity surface for usage, cost, workspace, and key management. Implement after the execution core is stable.

Compatibility rule: /v1/messages should be treated as a router, not a mere forwarder. In steady state it may send work to Anthropic, our own orchestrator, or a hybrid pipeline. The client should not need to know which path was chosen.

4. DSCO native API

This is the real backend surface: swarm-aware, topology-aware, async-first, and artifact-native.

5. Control plane topology

The right internal split is not a monolith pretending to be a proxy. It is a narrow edge plus explicit internal services.

EEdge gateway

Terminates TLS, authenticates keys, injects request IDs, preserves streaming semantics, and routes by namespace. This is the only public entrypoint.

RCompat router

Owns Anthropic-compatible translation. It decides whether a compatibility call is proxied upstream, internally executed, or hybridized.

PPolicy engine

Central place for workspace policies: model allowlists, MCP allowlists, tool budgets, timeout ceilings, and secret handling rules.

SSwarm scheduler

Allocates agents, enforces parallelism ceilings, tracks cancellations, and decouples execution lifetime from HTTP connections.

TTopology planner

Builds dynamic execution graphs from task profiles, prior outcomes, artifact retrieval, and runtime constraints.

AArtifact intelligence plane

Ingests critical files, chunks them, embeds them with Jina v4, and exposes retrieval to runs, planners, and audit tooling.

Do not put execution scheduling in the proxy handler. The edge should route and stream. The run service and scheduler should own state transitions, retries, heartbeat logic, and cancellation semantics.

6. Artifact and embedding plane

The artifact plane is where this stops being a stateless transport clone and becomes a system with memory.

IArtifacts to ingest first
  • ~/.claude.json
  • ~/.claude/CLAUDE.md
  • ~/.claude/debug/latest
  • recent ~/.claude/debug/*.txt
  • recent ~/.claude/backups/.claude.json.backup*
  • recent ~/.claude/file-history/**
  • ~/.claude/projects/**/memory/*.md
  • workspace AGENTS.md, CLAUDE.md, README*
EEmbedding strategy

Use jina-embeddings-v4 via Jina AI for all critical artifact chunks. Store normalized float32 vectors in SQLite blobs. Start with exact search. Introduce sqlite-vec only after the corpus size and latency profile justify it.

Artifact retrieval should be available to the topology planner, the swarm scheduler, review tooling, audit views, and MCP policy decisions.

{
  "artifact": {
    "artifact_id": "sha256(source)",
    "kind": "file | url | transcript | snapshot | config",
    "source": "absolute path or URI",
    "content_sha256": "dedupe key",
    "metadata_json": "{...}",
    "created_at": 0,
    "updated_at": 0
  },
  "artifact_chunk": {
    "artifact_id": "fk",
    "chunk_index": 0,
    "chunk_text": "...",
    "embedding_model": "jina-embeddings-v4",
    "embedding_dim": 0,
    "embedding_blob": "float32 blob"
  }
}

Important operational rule: raw artifact persistence should be configurable. Embeddings and metadata are broadly useful. Full-body storage should be policy-controlled because configs and logs can contain credentials or transport metadata you do not want retained in plaintext.

7. Streaming and execution rules

This is where previous failures happen if the backend is treated like ordinary request-response CRUD.

SSEStreaming requirements
  • Support long-lived SSE and chunked HTTP bodies without default 30-second watchdogs.
  • Preserve upstream headers such as x-api-key, anthropic-version, and anthropic-beta.
  • Do not treat repeated output patterns as corruption by default. Tables, JSON, and tool logs are normal.
  • Always emit a request ID and run ID so streaming failures can be resumed or replayed.
RUNRun semantics
  • Every long-running operation maps to a durable run record.
  • POST starts work and returns immediately when appropriate.
  • GET /ds/v1/runs/{id} reports status, reason, and partial progress.
  • Cancellation, detachment, and continuation are explicit state transitions, not side effects of client disconnects.

Hard requirement: never let edge timeouts kill execution state. At worst, disconnect the client stream and leave the run alive. Any design that couples the run lifetime to the socket lifetime will recreate the same failure mode.

9. Reference request flows

These are the flows that matter most when the client is pointed at api.distributed.systems.

F1Claude-compatible message flow
1

Client sends POST /v1/messages with Anthropic headers and a streaming preference.

2

Edge authenticates, assigns request ID, and passes the request to the compat router.

3

Compat router checks policy, model mapping, and whether the request should run upstream, internally, or as a hybrid route.

4

If internal execution is selected, a DSCO run is created and streamed through an SSE bridge back to the client.

5

All progress, tools, artifacts, and errors are captured into run_event independently of client socket health.

F2Dynamic swarm flow
1

Client sends POST /ds/v1/topologies/run or a compatible message request with swarm intent.

2

Topology planner classifies task profile and retrieves relevant artifacts from the embedding index.

3

Planner synthesizes a runtime graph with replica counts, validators, and feedback edges.

4

Swarm scheduler launches agents and records active stage state as the run progresses.

5

Partial outputs stream to the client, while the run remains resumable and queryable after disconnect.

10. Rollout plan

The order matters. Build the stable edge first, then deepen the control plane behind it.

P1Phase 1
  • /v1/messages
  • /v1/messages/count_tokens
  • /v1/models
  • /v1/mcp_servers
  • /v1/mcp/*
  • /api/event_logging/v2/batch
  • /ds/v1/artifacts/*
  • /ds/v1/runs/*
P2Phase 2
  • message batches
  • files compatibility surface
  • workspace and key management
  • usage and cost reporting
  • MCP policy and health views
P3Phase 3
  • full analytics parity
  • regional routing and failover
  • policy-driven execution brokering
  • artifact-aware replanning during active runs
  • tenant-specific retrieval and budget enforcement

Recommended immediate next implementation step: define the OpenAPI contract for /v1/messages, /v1/mcp/*, /api/event_logging/v2/batch, and the core /ds/v1/runs plus /ds/v1/artifacts endpoints before adding more runtime code. The edge contract needs to stop moving first.

This document is a design note for the backend shape behind api.distributed.systems. It assumes a compatibility facade for Anthropic and Claude traffic, a DSCO-native control plane under /ds/v1/*, SQLite-backed artifact and run state, and Jina AI jina-embeddings-v4 for retrieval over critical operational artifacts.
